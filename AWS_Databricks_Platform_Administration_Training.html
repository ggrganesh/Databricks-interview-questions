1. Introduction to Databricks

Before diving into administration, let's understand what Databricks is, why it has become so popular, and how it fits in the modern data stack.

1.1 What is Databricks?

Databricks is a unified, cloud-based data analytics and AI platform built on top of Apache Spark, Delta Lake, and MLflow — all originally created by the founders of Databricks.

It provides a single environment for:

  • Data engineering — ETL/ELT, batch + streaming ingestion
  • Data warehousing — SQL analytics over Delta tables
  • Data science & machine learning — model training, MLflow, serving
  • Business intelligence — dashboards, alerts, embedded analytics
  • Governance — unified security and lineage via Unity Catalog

Databricks runs on AWS, Azure, and GCP. This guide focuses on AWS.

1.2 Why Organizations Use Databricks

2. AWS Databricks Architecture

Understanding the architecture is the foundation of every security, cost, and networking decision an admin makes.

2.1 Control Plane vs Compute Plane

PlaneRuns InContains
Control PlaneDatabricks-managed AWS accountWeb UI, REST APIs, scheduler, notebook source, cluster mgr, UC metastore service, audit
Classic Compute PlaneCustomer AWS accountEC2 driver + worker nodes processing data
Serverless Compute PlaneDatabricks-managed AWS accountManaged compute for SQL warehouses, jobs, model serving
Important: With Classic compute, your data never leaves your AWS account. Only commands, metadata, and notebook source code flow through the Control Plane.

2.2 Two-Plane Architecture Diagram

Figure 2.1 — Two-plane architecture: metadata in Databricks' cloud; data in your AWS account.

2.3 AWS VPC

OptionDescriptionRecommended For
Databricks-managed VPCDatabricks provisions itPOCs, small teams
Customer-managed VPC (BYOVPC)You bring your own VPCProduction / Enterprise

Minimum VPC requirements:

  • Two private subnets in two AZs
  • A NAT Gateway (or fully air-gapped via PrivateLink)
  • Proper route table
  • DNS resolution + DNS hostnames enabled
  • CIDR range from /25 to /16 (more IPs = larger clusters)

2.4 EC2 Compute

Clusters are EC2 instances inside your VPC. Each cluster has one driver and N workers, plus local SSD for shuffle/cache.

2.7 Security Groups

  • Inbound: all TCP within the SG (intra-cluster)
  • Outbound: HTTPS (443) to Databricks, STS, S3 endpoints
  • No inbound from the internet

2.8 Networking Cheat Sheet

Recap — Section 2
  • Two planes: Control (Databricks-managed) and Compute (yours).
  • Data stays in your S3; only commands/metadata leave.
  • Customer-managed VPC is the enterprise default.
  • IAM roles + instance profiles + security groups are foundational.
  • PrivateLink provides fully private control plane connectivity.

4.5 Autoscaling & Auto-Termination

  • Scales up within seconds when there is backlog
  • Scales down after a delay when idle
  • Set auto-termination to 10–30 minutes on interactive clusters

1 driver (i3.xlarge)  = 0.75 DBU/hr x $0.55 = $0.41/hr
4 workers (i3.xlarge) = 4 x 0.75 DBU/hr x $0.55 = $1.65/hr

5. Unity Catalog Administration

5.1 What is Unity Catalog?

Unity Catalog (UC) is the unified governance solution for all data and AI assets in Databricks. It provides:

  • Centralized access control across all workspaces in a region
  • Fine-grained permissions (table, column, row, view)
  • Automatic data lineage
  • Data discovery + search
  • Built-in auditing
  • Native Delta Sharing
  • Governance for ML features and models

5.2 Unity Catalog Hierarchy

Figure 5.1 — Unity Catalog hierarchy

5.3 Metastore, Catalog, Schema

  • Metastore: top-level container. One per region (recommended).
  • Catalog: groups schemas. Common patterns: dev/qa/prod or by domain.
  • Schema: logical grouping of tables, views, volumes, functions.

5.4 Tables, Views, Volumes

5.5 External Locations & Storage Credentials

CREATE STORAGE CREDENTIAL my_storage_cred
  WITH (AWS_IAM_ROLE 'arn:aws:iam::123456789012:role/uc-storage-role')
  COMMENT 'Role used by UC to access prod S3';

CREATE EXTERNAL LOCATION sales_ext
  URL 's3://mycompany-sales-prod/'
  WITH (STORAGE CREDENTIAL my_storage_cred)
  COMMENT 'Sales data lake';

GRANT READ FILES, WRITE FILES
  ON EXTERNAL LOCATION sales_ext TO `data-engineers`;

5.6 Grants — Full SQL Reference

-- Catalog and schema access
GRANT USE CATALOG ON CATALOG prod              TO `data-analysts`;
GRANT USE SCHEMA  ON SCHEMA  prod.sales        TO `data-analysts`;

-- Table-level
GRANT SELECT ON TABLE  prod.sales.customers    TO `data-analysts`;
GRANT SELECT ON SCHEMA prod.sales              TO `data-analysts`;
GRANT MODIFY ON TABLE  prod.sales.customers    TO `data-engineers`;

-- Volumes / external locations
GRANT READ VOLUME, WRITE VOLUME
  ON VOLUME prod.ml.training_data              TO `ml-team`;
GRANT READ FILES, WRITE FILES
  ON EXTERNAL LOCATION sales_ext               TO `data-engineers`;

-- Revoke
REVOKE SELECT ON TABLE prod.sales.customers FROM `data-analysts`;

-- Inspect
SHOW GRANTS ON TABLE prod.sales.customers;

5.7 Row- and Column-Level Security

-- Column masking
CREATE FUNCTION mask_email(email STRING)
RETURNS STRING
RETURN CASE
  WHEN is_account_group_member('pii-readers') THEN email
  ELSE regexp_replace(email, '(^.)(.*)(@.*)', '$1***$3')
END;

ALTER TABLE prod.sales.customers
  ALTER COLUMN email SET MASK mask_email;

-- Row filter
CREATE FUNCTION us_only_filter(region STRING)
RETURNS BOOLEAN
RETURN region = 'US' OR is_account_group_member('global-readers');

ALTER TABLE prod.sales.customers
  SET ROW FILTER us_only_filter ON (region);

5.8 Data Lineage

Unity Catalog automatically tracks lineage between tables, notebooks, jobs, dashboards — including column-level lineage for SQL transformations. Visualize in Catalog Explorer → Lineage tab.

5.9 Governance Best Practices

  • One metastore per region
  • Separate catalogs per environment
  • Grant to groups, not users
  • Use service principals for prod pipelines
  • Use external locations, not legacy mounts
  • Enable system.access.audit and system.billing.usage
  • Document each catalog/schema with COMMENT
Recap — Section 5
  • UC = modern, unified governance.
  • 3-level namespace: catalog.schema.table.
  • External locations + storage credentials replace legacy mounts.
  • Lineage + audit + column/row masking are built in.

6.8 Audit Logs

Configure audit log delivery to an S3 bucket. Logs capture logins, cluster lifecycle, notebook commands, permission changes, UC ops, job runs. Ship to a SIEM (Splunk, Datadog) with ≥ 1-year retention. You can also query system.access.audit.

6.9 Security Best Practices

  • Enforce SSO + MFA
  • Enable SCIM for lifecycle
  • Use service principals for production
  • Disable / limit PATs
  • Enable IP access lists
  • Use PrivateLink (front-end + back-end)
  • Customer-managed VPC + KMS CMK
  • Secrets in AWS Secrets Manager-backed scopes
  • Ship audit logs to SIEM
  • Apply least privilege in UC grants
Recap — Section 6
  • Auth = who; Authz = what.
  • SSO + SCIM is non-negotiable for enterprise.
  • PrivateLink + customer VPC + CMK = enterprise-grade isolation.
  • Audit logs must be enabled from day one.

terraform {

provider "databricks" {
  host  = var.databricks_host
  token = var.databricks_token
}

resource "databricks_group" "engineers" {
  display_name = "data-engineers"
}

resource "databricks_cluster_policy" "etl" {
  name       = "etl-policy"
  definition = jsonencode({
    "spark_version"          : { "type": "fixed", "value": "15.4.x-scala2.12" },
    "node_type_id"           : { "type": "allowlist", "values": ["i3.xlarge"] },

    "autoscale.max_workers"  : { "type": "range", "maxValue": 8 }
  })
}

resource "databricks_permissions" "etl_policy" {
  cluster_policy_id = databricks_cluster_policy.etl.id
  access_control {
    group_name       = databricks_group.engineers.display_name
    permission_level = "CAN_USE"
  }
}

resource "databricks_catalog" "prod" { name = "prod" }

resource "databricks_grants" "prod_catalog" {
  catalog = databricks_catalog.prod.name
  grant {
    principal  = databricks_group.engineers.display_name
    privileges = ["USE_CATALOG"]
  }
}

12. Admin Best Practices

12.1 Security

  • Enforce SSO + MFA
  • SCIM for lifecycle automation
  • Service principals for production
  • IP access lists + PrivateLink
  • Customer VPC + KMS CMK
  • AWS Secrets Manager-backed scopes
  • Disable / strictly limit PATs
  • Audit logs → SIEM, ≥ 1-year retention
  • Least privilege in all UC grants

12.2 Cost Optimization

  • Photon for SQL/ETL
  • Job clusters for scheduled jobs
  • Auto-termination 10–30 min on interactive
  • Tag every cluster, job, warehouse
  • Cluster policies to cap instance size
  • Spot instances for non-critical workloads
  • Serverless SQL for BI
  • Weekly usage review + budget alerts

12.3 Governance

  • One metastore per region
  • Catalog per environment
  • External locations, not mounts
  • Grant to groups, not users
  • Column- / row-level security as needed
  • Track lineage for compliance
  • Enforce naming conventions

12.4 Performance

  • Photon-enabled clusters
  • Proper partitioning or liquid clustering
  • OPTIMIZE + VACUUM regularly
  • Autoscaling
  • Delta cache (i3 instances)
  • Right-size instance types

12.5 Operations

  • IaC (Terraform) from day one
  • DABs for CI/CD
  • Incident runbook
  • Documented on-call rotation
  • Quarterly access reviews
  • Monthly review of abandoned resources

15. Final Summary

15.1 Key Takeaways

  • Databricks is a unified Lakehouse on Spark + Delta + MLflow.

  • SSO + SCIM + service principals are required for production identity.
